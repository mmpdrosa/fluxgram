---
title: Flows
description: Define reusable flow trees from plain step descriptors and dynamic functions.
---

A flow is a named tree of steps. Fluxgram persists paths into that tree, so a chat can wait at a prompt or timer and later resume from the same position.

## Start With A Real Flow

```ts
const checkout = fx.flow("checkout", [
  send("Let's start checkout."),
  prompt.text("Email?", { store: "email" }),
  prompt.buttons("Shipping speed?", {
    buttons: [
      btn("Standard", set("shipping", "standard")),
      btn("Express", set("shipping", "express")),
    ],
  }),
  (ctx) => send(`Receipt will go to ${ctx.store.email}. Shipping: ${ctx.store.shipping}.`),
]);

fx.command("checkout", checkout);
```

The array is normalized into a durable flow tree. The prompt path is stored when the flow waits; the next reply or button click resumes from that path.

## Paths Through The Tree

Fluxgram stores path indexes, not JavaScript closures. In the checkout flow, the email prompt and shipping prompt have stable positions in the tree. If the process restarts after the email question, durable storage still knows which path is waiting.

That durability is why flow shape matters. Adding a step before an active prompt changes the path. For production deploys, use explicit versions when you change the shape of a flow that may already be active.

```ts
fx.flow("checkout", checkoutSteps, { version: 2 });
```

## Dynamic Steps

Functions are dynamic steps. They receive `ctx`, can read `ctx.store`, and may return another step or array of steps.

```ts
fx.flow("dynamic", [
  prompt.text("Age?", { store: "age" }),
  (ctx) => {
    const age = Number(ctx.store.age);
    if (age < 18) return send("A guardian must continue this flow.");
    return send("You can continue.");
  },
]);
```

Dynamic steps are useful for small decisions close to the conversation. For larger branches, use `branch` so the shape is easier to see.

## Store

`ctx.store` is persisted with the flow state. Prompt `store` options write into it, `set(...)` writes explicit values, and programmatic starts can seed it.

```ts
await fx.initiateFlow("checkout", chatId, {
  store: { cartId: "cart_123" },
});
```

Keep store values JSON-compatible when using durable adapters. Fluxgram enforces this:
non-JSON store values (Dates, Maps, class instances, functions, `NaN`) are rejected
before persisting, with a path-pointed error. The check is on outside production; set
`validateStoreJson: false` to disable it (or `NODE_ENV=production`, where it is off by
default).

The store is untyped by default. For opt-in compile-time checks on store keys and
value types, see [Typed Stores](/docs/typed-stores) — and note that store keys are
never proven to be _present_ (reads are `T | undefined`; guard them).

## Safe Versioning Patterns

- Safe: change copy in `send(...)` or prompt labels without changing structure.
- Usually safe: append new steps after a point where no old flow can currently wait.
- Risky: insert, remove, or reorder steps before active prompts or timers.
- Safer deploy: register a new explicit `version` and choose a `versionMismatch` policy.

The core rule: if a conversation may be waiting inside a flow, treat its tree shape as persisted data.
