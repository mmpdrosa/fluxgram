`Fluxgram` is the grammY-facing runtime. It registers flows, wires Telegram updates into the engine, and owns the rules for routing, queueing, lifecycle, and active-flow administration.

```ts
const fx = new Fluxgram(bot, {
  storage: new MemoryStorage(),
  onFlowError: (ctx) => send(`Recovered: ${String(ctx.error)}`),
});
```

## Update Flow

Incoming messages move through these checks in order:

1. Group migration service messages are recorded for outbound chat-id sanitization and active-flow alias lookup.
2. `cancelCommand` and `overrideActive` commands pre-empt active prompts.
3. Waiting prompts get priority over normal commands and `onMessage` handlers.
4. Registered commands start their flows.
5. Message triggers run only when no prompt handled the update.

That priority keeps conversation state predictable: while a prompt waits, the next relevant message answers the prompt unless you explicitly register a reset command.

## Commands And Reset Patterns

```ts
fx.command("start", "start");
fx.commands(["menu", "home"], "menu", { overrideActive: true });
fx.cancelCommand("cancel", {
  text: "Cancelled.",
  notActiveText: "Nothing to cancel.",
});
```

`overrideActive` kills active or waiting flows in the chat before the command runs. Use it for reset commands like `/start`, `/cancel`, or `/clear`.

## Message Triggers

```ts
fx.onMessage({ regex: /support/i }, fx.flow("support", [send("How can we help?")]));
fx.onMessage({ filter: (msg) => msg.photo !== undefined }, "photo-review");
```

Plain message triggers are fallback routing. They run only when no flow is waiting for that chat.

## Middleware

```ts
fx.use(async (ctx, next) => {
  if (ctx.chatId < 0 && ctx.source === "command:/start") {
    return ctx.block();
  }
  ctx.params.store = { source: ctx.source };
  return next();
});
```

Middleware can block a start, set initial store data, or replace the flow before it starts. Sources can target all flows, commands, messages, or programmatic starts.

## Per-Chat Queueing

Fluxgram serializes execution per chat and allows different chats to run concurrently. This prevents double-click and waiter cleanup races while keeping independent chats responsive.

Durable timers release the queue while they wait. Active async steps hold the queue until they finish.

## Runtime Lifecycle

```ts
await fx.init();
await fx.sweep();
await fx.start();
await fx.stop();
```

`start()` calls `init()`, starts the recovery sweep interval, and starts grammY long polling. Use `init()` directly when another server routes updates and you need Fluxgram ready before handling them; see [Webhook Deployments](/docs/webhooks) for the full webhook pattern.

## Active Flow Inspection

```ts
const active = await fx.listActive({ chatId });
const handle = active[0] ? await fx.getFlowHandle(active[0].id) : null;
await fx.gcFlows(active.map((item) => item.id));
```

Use active flow inspection for admin tools, support dashboards, cleanup jobs, and cancel/reset behavior that is more specific than `cancelCommand`. Chat-scoped inspection follows known group migration aliases, so querying the current supergroup id also finds flows that started before migration.

See [Operations](/docs/ops) for copyable active-flow, handle, cleanup, broadcast, `DebugChatSink`, and `FluxgramClient` patterns.

## Group Lifecycle

```ts
fx.onAddedToGroup("group-setup");
fx.onBecameAdmin("admin-setup");
fx.onLostAdmin("admin-lost");
fx.onGroupMigrated("group-migrated");
```

Fluxgram tracks group-to-supergroup migrations. `sanitizeChatId` resolves current chat ids, and `getOriginalChatId` can map a migrated supergroup id back to the original id when known.

## Hooks And Client Events

```ts
fx.onBotSentMessage((message) => console.log(message.message_id));
fx.onBotHandledMessage((message) => console.log(message.chat.id));
```

`FluxgramClient` can send cross-process events through an event bus for actions such as `sendMessage` and `initiateFlow`.
