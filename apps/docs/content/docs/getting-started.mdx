---

This guide builds a small bot that asks for a name, waits for the next message, stores the answer, and resumes the flow.

## Install

```sh
bun add fluxgram grammy
```

## Create The Bot

```ts
import { Bot } from "grammy";
import { Fluxgram, MemoryStorage, prompt, send } from "fluxgram";

const bot = new Bot(Bun.env.BOT_TOKEN!);
await bot.init();

const fx = new Fluxgram(bot, {
  storage: new MemoryStorage(),
});
```

`bot.init()` gives Fluxgram the bot identity it uses to namespace durable state. `MemoryStorage` is fine for local development and tests; use durable storage before production.

## Register The First Flow

```ts
fx.command(
  "start",
  fx.flow("start", [
    send("Welcome. I can remember where you are in this conversation."),
    prompt.text("What should I call you?", { store: "name" }),
    (ctx) => send(`Nice to meet you, ${ctx.store.name}.`),
  ]),
);
```

The command starts a named flow. The text prompt sends a message and stores a waiter. The dynamic final step runs only after the user replies.

## Run The Bot

```ts
await fx.start();
```

`start()` calls `init()`, starts recovery sweeps, and starts grammY long polling. If you deploy behind webhooks instead, call `await fx.init()` before routing updates and schedule `await fx.sweep()` yourself.

## What Happened At Runtime

1. `/start` matched the command and created a flow state for the chat.
2. `send(...)` posted the welcome message.
3. `prompt.text(...)` posted the question, persisted the waiting path, and stopped the flow.
4. The user's next text message claimed the waiter, stored `ctx.store.name`, and continued from the next step.

While a prompt is waiting, messages in that chat answer the prompt before normal command or message-trigger routing. Use `overrideActive` or `cancelCommand` for reset commands.

```ts
fx.command("cancel", fx.flow("cancel", [send("Cancelled.")]), {
  overrideActive: true,
});
```
